# 研發流程優化與團隊效能提升建議報告

**提交人：** Staff Front-end Engineer  
**主旨：** 針對甘特圖管理侷限、進度檢討成本與研發效能之改進建議

---

## 一、 現況核心問題：管理手段與產出目標的背離

目前研發團隊陷入「為了更新進度而無法工作」的困境。核心痛點在於：

1. **甘特圖的剛性預測與現實脫節：** 甘特圖假設開發是「線性且無變數」的，但軟體開發充滿不確定性（如 API 延遲、需求異動）。目前「強行壓縮」且「缺乏修正空間」的排程，導致工程師被迫犧牲 PR 與 UT 質量，直接造成目前 **100+ 個 Bug** 的積壓。

2. **「進度檢討會」產生的負向產能：**
   當甘特圖進度因合理因素（如品質維修、上游延誤）落後時，目前的管理反應是「叫入辦公室開會」。
   * **成本損失：** 每次中斷開發進入會議，工程師需花費 **20-30 分鐘** 才能重新進入深度工作（心流）。
   * **惡性循環：** 為了檢討「落後」，卻消耗了更多本該用於「追趕」的開發時間，導致進度更落後。

3. **溝通與獎金機制失靈：**
   LINE 溝通碎片化與個人/團體獎金的二元對立，使工程師在「保命（修 Bug 領錢）」與「保全專案（如期上線）」之間產生心理撕裂，最終導致「理性躺平」。

---

## 二、 系統性改進建議（依急迫性排列）

### 1. 第一階段：降低行政損耗（即刻實施）

| 改進項目 | 具體做法 | 預期改善 |
| :--- | :--- | :--- |
| **減少實體檢討會** | 除非甘特圖停滯超過 3 日，否則改由 **Discord/Teams 異步更新**，禁止動輒叫進辦公室。 | 釋放每日至少 1-2 小時的核心開發產能。 |
| **DRI 權責鎖定** | 確立功能模組的 DRI，QA 報錯直接對接負責人。 | 減少「問來問去」的無效會議與溝通成本。 |
| **階層式團體獎金** | 設立「Buffer Week（修正週）」，此週專門處理 Bug，不排新進度，過關仍發 50% 獎金。 | 激發團隊互助清空 100+ Bug，而非各掃門前雪。 |

### 2. 第二階段：時程控管優化（針對 3/27 專案）

| 改進項目 | 具體做法 | 預期改善 |
| :--- | :--- | :--- |
| **甘特圖「滑動緩衝」** | 依賴項目（如 API）延遲，下游前端開發時程應**自動順延**，嚴禁壓縮後端測試時間。 | 確保 PR/UT 有 20-30% 法定時間，攔截低級 Bug。 |
| **溝通工具分層** | 規格討論、進度報備移至 **Google/Teams**。LINE 僅限緊急聯絡。 | 資訊分層，使工程師能「非同步」處理非緊急任務，保障專注力。 |
| **有效工時評估** | 認可開會、溝通、PR 審核也是工作。甘特圖僅以**每日 5-6 小時**開發量排程。 | 提高時程表的可信度，避免「虛假的準時」導致上線崩潰。 |

## 三、 財務與成本考量（資方說服點）

老闆最在意的「產出」與「加班價值」分析：

1. **會議的代價：** 若 5 位工程師開會 1 小時，加上心流恢復時間，公司實際損失的是 **7.5 個高薪工時**。若頻繁開會，每月浪費的薪資成本高達數萬元，且未產生任何代碼產出。

2. **Bug 獎金的槓桿迷思：** 老闆支付 Bug 修正獎金，實際上是在為「當初壓縮工時」買單。**與其花錢買「修補」，不如透過保障 PR/UT 時間買「穩定」。** 降低 Bug 數量，直接等於降低人力成本。

3. **軟體資產安全性：** 目前的「緊上加緊」讓 PR/UT 形同虛設。3/27 訂單系統若出現資安或邏輯漏洞，其修復與客訴賠償成本將是現在「調整管理方式」的 **100 倍以上**。

---

## 四、 結論

**「過度管理會殺死產能。」**
頻繁的進度檢討會與僵化的甘特圖，正將團隊推向躺平。為了確保 3/27 專案成功，我們需要的是**「釋放開發時間」**與**「建立合理的容錯/修正緩衝」**。

建議老闆支持從 **LINE 轉向專業分層工具**，並承認 **PR/UT/開會** 是開發的一部分，這才是保護公司軟體資產、降低 Bug 數量的唯一出路。
